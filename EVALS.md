# 评估方法论(EVALS.md)

这份文档说明系统的评估方式,对应 FDE 岗位面试里常被问到的问题:"怎么证明你的 Agent 上线后不会跑偏/不会乱编?"

## 为什么需要评估,而不只是"试了几次感觉不错"

任何 RAG/Agent 系统的演示效果都可能是挑出来的几个好案例。真正能证明系统可靠性的,是一份**固定、可重复、覆盖正反两类场景**的测试集,以及跑出来的可量化结果——这就是 evals(评估工程),也是这个项目区别于"跑通了教程的 hello world"的地方。

## 测试集设计

在 `backend/eval_data.py` 中,固定了一份贴近真实客服场景的企业服务类测试文档(SaaS 产品的价格、政策、技术规格等),配套 24 道金标准测试题,分三类:

- **15 道可回答题**:文档里确实有明确答案,检查系统能否准确找到并复述关键信息。
- **5 道不可回答题**:文档里完全没提到的内容(如"CEO是谁"),检查系统是否会老实说不知道。
- **4 道诱导编造陷阱题**:问法故意设计得像有答案("基础版客服机器人支持几种语言"——文档只说了专业版,没说基础版),专门测试模型会不会被诱导着把相邻信息错配成答案。这类题是最容易被纯粹"看起来智能"的系统翻车的地方。

## 打分逻辑与一个真实发现的设计问题

可回答题:回答或引用片段中只要命中预期关键词任意一个就算通过,不要求逐字匹配(因为模型会用自己的话复述)。

不可回答题:回答中出现拒答类措辞,或 `grounded` 字段为 `false`,判定为正确拒答。

**开发过程中这里发现过一个值得记录的设计问题**:纯检索模式(未配置 API Key 时)本质上没有语义判断能力——它只会把 BM25 命中度最高的内容甩出来,不具备"判断这段内容是否真的回答了问题"的能力,所以拿不可回答题去考核它是不公平的,会得出一个看起来很差但其实是设计使然的分数。修正后的逻辑是:**纯检索模式只评估"可回答题的检索召回率",不评估拒答能力**;拒答能力只在配置了 API Key 的智能问答模式下才纳入评分,并在报告里显式标注原因,而不是显示一个误导性的低分。

## 怎么看结果

打开 `/evals.html`,点击"运行评估",会真实跑一遍完整链路(建会话、上传测试文档、逐题提问),返回:总通过率、可回答题命中率、拒答准确率(纯检索模式下显示为不适用并说明原因)、每道题的详细结果和失败原因。这是端到端评估,不是单元测试——跑的是和用户实际使用完全一样的路径。

## 当前基线(纯检索模式)

可回答题命中率 100%(15/15),拒答类指标因模式限制不适用。配置 API Key 切换到智能问答模式后,可以拿到完整的 24 题评分,包括最关键的"诱导编造陷阱题"通过率——这个数字是面试时最值得主动展示的证据。

## 已知局限

测试集是固定的 24 题,数量不大,无法覆盖所有边界情况。生产环境的做法是持续把真实用户问到、系统答错或答不上来的 case 沉淀进测试集,让 golden set 跟着真实使用慢慢变大——这正是 FDE 岗位强调的"产品反馈闭环"。
